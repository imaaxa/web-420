Explain the 7-step design procedure.

  1. List the Semantic Descriptors: This is a list of everything that might go in or out of the API.
  2. Draw a state diagram: Use boxes to represent a representation, a goup of semantic descriptor, arrows show state transitions.
  3. Reconcile Names: Try to match up your needed semantic needs with a pre-existing progils, IANA-registered links.
  4. Choose Media Type: Choose a media type that is compatible with your protocol semantics.
  5. Write a Profile: Write a profile that documents your application semantics.
  6. Implementation: Write your application's code.
  7. Publication: Publish your API documentation, tutorials, and examples.

* From O'Reilly RESTful Web API's, Chapter 9: The Design Procedure.

Explain the versioning process for media types and profiles.

"A REST API should spend almost all of its descriptive effort in defining the media type(s) used for representing resources and driving application state, or in defining extended relation names and/or hypertext-enabled mark-up for existing standard media types. (Paraschiv, 2019)"

The two main types of versioning are in relation to URI Versioning, and Media Type Versioning.

When we develope an API around a URI space, new URI space or a new URI version is needed. This is due to the immutableality of the representations of resources.

When we develope an API around custom media types, we need to version it according to media type versioning.

References:

Paraschiv, E. (2019, October 18). Versioning a REST API. Retrieved November 20, 2019, from https://www.baeldung.com/rest-versioning.
